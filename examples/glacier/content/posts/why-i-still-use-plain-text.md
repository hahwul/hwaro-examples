I keep my notes in plain text. My todo lists are plain text. My journal is plain text. My blog posts are written in Markdown, which is plain text with minimal formatting. I have been doing this for over a decade, and I have never lost a file to a format migration, a discontinued service, or a corrupted database.

This is not a coincidence. It is the natural consequence of choosing the simplest format that meets the requirement.

## The Durability Argument

Every proprietary format is a ticking clock. The company that created it may be acquired, may pivot, may shut down. The application that reads it may drop support in a future version. The cloud service that hosts it may change its terms, raise its prices, or disappear entirely.

Plain text has none of these risks. A `.txt` file written on a computer in 1985 is perfectly readable today. No conversion, no import wizard, no subscription. Every operating system, every text editor, every programming language can read and write plain text. This will still be true in fifty years.

## The Portability Argument

Plain text files work everywhere. They sync with any file syncing service. They version-control with Git. They search with grep. They process with awk, sed, and every scripting language ever written. They back up by copying a directory.

Try doing any of these things with a Notion database, an Evernote notebook, or a Confluence space. You will find yourself writing export scripts, dealing with lossy format conversions, or locked into a vendor's ecosystem with no clean exit path.

## The Simplicity Argument

Plain text imposes no structure beyond what you choose to create. This is a feature, not a limitation. When a tool imposes structure, you must either conform to its model of how information should be organized or fight against it constantly.

With plain text, the structure is yours. A flat file of bullet points. A directory tree of topic-specific notes. A single long document with sections. Whatever matches how you think. And when your thinking changes, you restructure with `mv`, `cat`, and a text editor. No migration, no schema change, no API call.

## The Tools Argument

The tools for working with plain text are the best tools in computing. Vim and Emacs are more powerful than any rich-text editor. grep and ripgrep search faster than any database-backed full-text search. Git provides version history that no note-taking app can match. These tools have been refined over decades by millions of users. They are fast, reliable, and deeply customizable.

## What You Give Up

Formatting. Embedded images. Real-time collaboration. WYSIWYG editing. These are real losses, and for some use cases, they matter enough to justify a different tool.

But for the core activity of capturing, organizing, and retrieving your own thoughts, plain text is the right answer. It has always been the right answer. Every few years, a new tool tries to convince us otherwise, and every few years, the people who chose plain text are still quietly productive while others are migrating their data to the next shiny thing.

Choose the format that will outlast the tools. Choose plain text.
