How it works

  1. 1

    Describe a property

    Instead of one example, state a rule that should hold for every input a generator can produce — sorting stays sorted, encoding round-trips, a queue never loses an item.

  2. 2

    Apatite hunts for a break

    It generates hundreds of random cases from your generators — integers, lists, records, command sequences — and stops the instant one falsifies the property.

  3. 3

    It shrinks to minimal

    The failing case is reduced, step by step, to the smallest input that still reproduces the bug — and a seed is printed so you can replay that exact case forever.

Why Apatite

Shrinking is not optional

Every generator Apatite ships — integers, lists, strings, records, even command sequences — knows how to shrink itself. You never write a shrinker by hand.

Every failure replays

Each run prints the seed that produced it. Pin that seed in a regression test and the exact counterexample — not a similar one — runs again on every build.

Generators compose

Build a generator for a domain type once, out of map, filter, and oneOf, and every property that needs one just asks for it.
