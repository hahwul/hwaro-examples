Waypoint 1.4.0 is the release where lazy pulls stop being per-image and start
being per-chunk. Layers are now split into content-defined chunks, hashed, and
stored once per host — so the fourteenth Debian-based image on a node pulls
almost nothing. On our build fleet the average cold start fell from 340 ms to
180 ms, and registry egress dropped by 61 percent.

<!-- more -->

## Added

- **Content-defined chunking.** Layers are carved into variable-length chunks
  with a rolling hash, so a one-line change to a base image only invalidates the
  chunks that actually moved. Deduplication is global across every image in the
  local store, building on the per-file table of contents added in
  [v1.2.0](../v1-2-0/).
- **Peer-to-peer lazy pulls.** When several nodes share a subnet, Waypoint asks
  its neighbours for a chunk before falling back to the registry. Enable it per
  host:

```toml
# /etc/waypoint/waypoint.toml
[pull.p2p]
enabled = true
discover = "mdns"        # mdns | static
max_peers = 8
```

## Changed

- The prefetch predictor now trains on per-image access traces instead of a
  global model. First-read stalls on cold nodes are roughly halved.
- The chunk-store snapshotter is the default backend. Existing per-layer mounts
  keep working; pass `--snapshotter=fuse-layer` to opt back in.

## Fixed

- A race between the FUSE writeback queue and container teardown could leave a
  container blocked on I/O when hundreds started at once. The queue now drains
  before the mount is released.

## Upgrading

Pinned digests are unaffected. To adopt chunk dedup for images already on disk,
re-import them once:

```bash
waypoint image reindex --all
waypoint prefetch --profile=warm registry.example.com/api:1.4
```
