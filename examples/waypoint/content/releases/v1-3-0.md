Waypoint 1.3.0 focuses on the second container start, not the first. Once an
image has been read lazily once, you rarely want to pay that cost again — so
this release adds warm pools, checkpoint/restore, and prefetch profiles that
capture what "ready" means for a given workload.

<!-- more -->

## Added

- **Checkpoint / restore.** `waypoint checkpoint` freezes a container's memory
  and its lazy mount state into a single snapshot that restores on another host
  without re-reading the registry.

```bash
waypoint checkpoint api-1 --out api.warm
waypoint restore api.warm --name api-2
```

- **Warm pools.** Declare a pool and Waypoint keeps N containers materialized
  and paused, handing one over on request and refilling in the background.
- **Prefetch profiles.** A profile is a named glob set describing the files a
  workload touches on startup, so the predictor never has to guess:

```toml
[prefetch.profiles.web]
paths = ["/app/**", "/usr/lib/node_modules/**", "/etc/ssl/**"]
concurrency = 6
```

## Changed

- arm64 is now a release-blocking target. Multi-arch manifests are produced for
  every tag, and the test matrix runs on both architectures.

## Fixed

- Files materialized after container start could inherit the fetch time as their
  modification time, confusing build caches. The stored layer mtime is now
  preserved through the lazy path.
