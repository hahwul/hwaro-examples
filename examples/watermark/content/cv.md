## Summary

Technical writer specializing in developer-facing documentation: API reference, SDK quickstarts, authentication guides, and error taxonomies. Ten years across payments, identity, and messaging platforms. I work docs-as-code — Markdown or MDX in the same repo as the SDKs, reviewed in the same pull requests as the endpoints they describe — and measure success by whether an integration finishes without a support ticket.

## Experience

**Senior Technical Writer — Meridian Pay**
*2022 — present*

- Own the public API reference and four official SDKs (Go, Python, Node, Ruby) for a payments platform used by 400+ integrating teams.
- Introduced a single OpenAPI 3.1 source of truth that generates the reference docs, SDK method stubs, and Postman collection from one spec.
- Cut integration support tickets by roughly a third after rewriting the idempotency-key guide with runnable request/response pairs.

**Technical Writer — Anchorpoint**
*2019 — 2022*

- Rebuilt the authentication documentation from an internal wiki export into a versioned, docs-as-code reference with copy-pasteable examples in five languages.
- Wrote and maintained the OAuth 2.0 and token-refresh guides that became the most-visited pages in the developer portal.
- Ran a quarterly "docs audit" pairing writers with the engineers who shipped that quarter's API changes.

**Documentation Engineer — Tidewave**
*2016 — 2019*

- Designed the structured error-reference format — one taxonomy, one page per error family — still in use across Tidewave's messaging and notification APIs.
- Built the CI check that fails a pull request when an endpoint change ships without a corresponding docs diff.

**API Support Engineer — Fjord Cloud**
*2014 — 2016*

- Answered integration questions on the developer support queue; used the recurring questions to write the first version of the SDK quickstarts.
- Transferred to the newly formed documentation team after the quickstart rewrite cut ticket volume for new SDK users by half.
