A widely-used commercial cryptography library was discovered, in late April 2026, to be emitting random bytes with a small but measurable 7-bit bias. The library's authors have been notified and the issue is now public; this observation summarizes what we measured.

## the symptom

A team at a hardware-security-module vendor noticed that the keys they were generating from this library's `secure_random_bytes()` function exhibited unusually clean distributions in their conformance tests — *too* clean. The team performed a chi-squared test on 4 GB of output and found that the seventh bit position was overrepresented by approximately 1.8% relative to a true uniform distribution.

This is, on its face, a small bias. It is well within the noise floor of most random-quality tests. Dieharder does not flag it. The NIST SP 800-22 suite does not flag it. It is invisible unless you specifically test the seventh bit position across a large sample, which is not a standard part of any of the major suites.

## the cause

We obtained source code access through the vendor's responsible-disclosure channel. The library's RNG is built on a CSPRNG that wraps a hardware random number generator. The hardware RNG produces 64-bit outputs in batches of 16. The library re-orders the bytes for endianness compatibility with the consumer's expected byte order.

The endianness re-ordering routine was subtly incorrect. In the affected version, byte 7 of every 16-byte block was being XORed with a constant value derived from a stack variable that was, on this implementation, almost always the same. The XOR did not destroy the random property entirely — the underlying byte was still random — but it shifted the distribution of bit 7 in that byte by exactly the amount we measured.

The bug had been in the library since 2022. It survived a major-version refactor in 2024 because the unit tests for the endianness routine compared byte-by-byte against an expected output, and the expected output had been generated by the buggy routine.

> The pattern of "the unit tests were generated by the buggy code" is one of the most insidious bug-survival patterns in cryptographic libraries. The tests pass. The library passes its own tests. Nothing is wrong, until you compare against an external source of truth — which, in this case, was a chi-squared test on a large sample of bit 7.

## the impact

The library is used in approximately 240 commercial products. Of those, 41 use it for key generation. The bias is *small enough* that most uses of the library remain secure in practice — a 256-bit key with a 1.8% bias on one of its 256 bits still has well over 250 bits of effective entropy. But the bias is *large enough* that it could be detected by an adversary with access to a large corpus of generated keys, and could be the seed of a more practical attack against specific use cases.

The most worrying use is in key-pair generation for ECDSA on the NIST P-256 curve. The bias does not on its own enable key recovery, but it does mean that the keys are not uniformly distributed on the curve. Combined with the well-known nonce-reuse vulnerability in some ECDSA implementations, the bias provides a small but real channel for key recovery.

## the fix

The library's authors released a patched version within 48 hours of being notified. The patched version emits keys that are uniformly distributed across all 256 bit positions. The vendor has issued an advisory recommending that keys generated with the affected versions be rotated.

## what we recommend

If you use any version of this library released between 2022-Q3 and 2026-Q2 for key generation:

1. **Read the vendor's advisory.** It is the canonical source.
2. **Identify the keys that were generated by the affected library.** Most production deployments do not record the library version used at key-generation time. We recommend you check by inspection of any documentation, build manifest, or recovery point.
3. **Rotate the keys.** The patched library produces uniformly distributed keys. Existing keys are not catastrophically compromised — most uses remain secure — but rotation is cheap insurance against future research that might find a more practical attack.
