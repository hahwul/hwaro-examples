This page covers the steps required to bring a Sentinel monitoring node online and route its first signal to the central collector.

## Prerequisites

A Sentinel node requires a host with at least two CPU cores, four gigabytes of memory, and outbound access to the collector endpoint over TCP port 7443. The agent runs as an unprivileged service account and must be granted read access to the log paths it is configured to watch.

The collector itself is delivered as a single static binary. It maintains its own persistent store and does not require an external database for deployments under fifty nodes.

## Installation

Download the agent package matching the host architecture and verify its checksum against the published signature. The checksum file is signed with the project release key, which is available from the documentation root.

The package installs to `/opt/sentinel` and registers a system service. The default configuration is loaded from `/etc/sentinel/agent.toml` and may be overridden by environment variables prefixed with `SENTINEL_`.

## First Signal

After installation, start the agent and confirm that it has registered with the collector. The collector exposes a registration endpoint that returns the current set of known nodes. A successful registration produces a single line in the agent log containing the assigned node identifier.

The first signal is generated by the heartbeat probe, which runs every fifteen seconds by default. The interval is configurable, but values below five seconds are rejected to prevent collector saturation.

## Next Steps

Once the agent is reporting, configure at least one log source and one metric source. The reference configuration in the appendix provides a tested starting point for common workloads. Custom probes are defined in the probe directory and loaded at agent startup.
