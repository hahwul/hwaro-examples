Several customers in the public sector and in regulated industries needed
to state, in writing, exactly which country their analytics data is
processed in — "somewhere in the EU" wasn't a good enough answer for their
auditors. 1.2 makes the region a first-class, enforced setting.

<!-- more -->

## Choosing a region

Region is now set per project, at creation time, from a dropdown with two
options: Frankfurt or Dublin. Once chosen, the setting is locked — we do
not support silently migrating a project's historical data between
regions, because a migration is itself a cross-border transfer and we
would rather refuse the feature than build it carelessly.

```toml
[project]
name = "example.eu"
region = "eu-central-1"   # Frankfurt. Use "eu-west-1" for Dublin.
```

## What "enforced end to end" means

The collection edge nearest to a visitor still receives the request first,
for latency, but it does no processing — it forwards the raw request over
an internal EU-only network directly to your project's pinned region and
does not persist anything locally. Hashing, aggregation, and storage all
happen in the pinned region and nowhere else. We had this working for
storage before 1.2; this release closes the gap at the edge.

## Existing projects

Projects created before 1.2 have been pinned to Frankfurt automatically,
matching where their data already lived. If you need Dublin instead,
open a support request and we'll create a fresh project ID for you — again,
by design, not a background migration.

## Compliance documentation

Pinned projects now generate a one-page residency statement from account
settings, suitable for attaching to a vendor security questionnaire,
listing the exact region, the legal entity operating it, and the retention
window in effect.
