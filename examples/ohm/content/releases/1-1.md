The browser script covers pageviews well, but a lot of the events that
matter to our customers — a subscription starting, a webhook firing, a
batch job finishing — never touch a browser at all. 1.1 adds a
server-side endpoint for exactly that case.

<!-- more -->

## The endpoint

```bash
curl https://api.ohm.eu/v1/events \
  -H "Authorization: Bearer $OHM_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "subscription_started",
    "domain": "example.eu",
    "props": { "plan": "growth" }
  }'
```

Events accept up to eight custom properties, all of which must be strings,
numbers, or booleans — nothing that could accidentally carry personal data
into a property value, because we do not scan payloads for that on your
behalf. Property keys and values are capped at 200 characters.

## Rate limits

The endpoint accepts 100 requests per second per project on Growth and
Scale, and 20 requests per second on Starter. Requests over the limit
receive a `429` with a `Retry-After` header rather than being silently
dropped, so a burst from a batch job degrades gracefully instead of losing
data.

## Dashboard changes

Server-side events appear in a new "Events" tile on the dashboard, counted
the same bare-numeral way as everything else — a total for the period and
a one-word label, no chart. Filtering by event name is available from the
project settings sidebar.

## Client libraries

We shipped a small Go client and a small Node client alongside the HTTP
API, both under 200 lines, both with zero dependencies beyond the standard
library's HTTP client. Neither does anything the `curl` example above
doesn't — they just save you writing the retry logic yourself.
