## Overview

The 2000 series covers application-layer and business logic errors. These codes are generated by the API gateway, request processors, and domain services.

---

## Error Code Reference

| Code | Severity | Description | Recommended Action |
|------|----------|-------------|--------------------|
| ERR-2000 | Error | Request validation failed | Check the request payload against the API schema |
| ERR-2001 | Error | Required field missing | Include all required fields in the request body |
| ERR-2002 | Warning | Deprecated field used | Migrate to the replacement field before the removal date |
| ERR-2010 | Error | Authentication required | Provide a valid authentication token in the Authorization header |
| ERR-2011 | Error | Authentication token expired | Refresh the token using the refresh token endpoint |
| ERR-2012 | Error | Authentication token malformed | Ensure the token is a valid JWT with the correct structure |
| ERR-2020 | Error | Authorization denied | The authenticated user lacks permission for this resource |
| ERR-2021 | Warning | Authorization policy evaluation slow | Complex policies are degrading performance; review policy rules |
| ERR-2030 | Error | Resource not found | The requested resource does not exist or has been deleted |
| ERR-2031 | Error | Resource conflict | The operation conflicts with the current state of the resource |
| ERR-2032 | Warning | Resource nearing quota limit | Usage is at 90% of the allocated quota |
| ERR-2033 | Critical | Resource quota exceeded | Usage has exceeded the allocated quota; requests are being rejected |
| ERR-2040 | Error | Rate limit exceeded | Reduce request frequency or request a higher rate limit |
| ERR-2041 | Warning | Rate limit threshold approaching | Current request rate is at 80% of the limit |
| ERR-2050 | Error | Payload too large | Reduce the request body size below the maximum of 10MB |
| ERR-2060 | Error | Unsupported media type | Use a supported content type (application/json) |
| ERR-2070 | Critical | Internal processing error | An unexpected error occurred; contact support with the request ID |
| ERR-2080 | Error | Downstream service unavailable | A required dependency is not responding; retry after a delay |
| ERR-2090 | Warning | Response serialization slow | Large response payloads are causing serialization delays |

## Detailed Descriptions

### ERR-2000: Request Validation Failed

The request body or query parameters do not conform to the expected schema.

**Log Example:**

```
2026-03-16 10:22:15 ERROR [api-gateway] ERR-2000
  endpoint=POST /api/v2/orders
  request_id=req-f5e6d7c8
  validation_errors=[
    {"field": "quantity", "error": "must be a positive integer"},
    {"field": "shipping_address.postal_code", "error": "invalid format"}
  ]
```

**Diagnostic Steps:**

1. Retrieve the API schema for the endpoint:

```bash
curl -s https://api.example.com/openapi.json | \
  jq '.paths["/api/v2/orders"]["post"]["requestBody"]'
```

2. Validate the request payload offline using a schema validator:

```bash
# Using ajv-cli
echo '{"quantity": -1, "shipping_address": {"postal_code": "abc"}}' | \
  ajv validate -s schema.json -d /dev/stdin
```

3. Check the server-side validation error response for details:

```bash
curl -s -X POST https://api.example.com/api/v2/orders \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"quantity": -1}' | jq '.errors'
```

### ERR-2040: Rate Limit Exceeded

The client has sent too many requests within the rate limit window.

**Log Example:**

```
2026-03-16 11:15:30 ERROR [rate-limiter] ERR-2040
  client_id=app-web-client
  endpoint=/api/v2/search
  window=60s
  limit=100
  current=103
  retry_after=42s
```

**Diagnostic Steps:**

1. Check the current rate limit headers in the response:

```bash
curl -s -D - https://api.example.com/api/v2/search \
  -H "Authorization: Bearer $TOKEN" 2>&1 | \
  grep -i "x-ratelimit"
```

Expected headers:

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1711018572
Retry-After: 42
```

2. Review the application code for unintentional request amplification (polling loops, retry storms):

```bash
# Search for tight loops or aggressive retry logic
grep -rn "while.*retry\|sleep(0)\|setInterval.*100" src/
```

3. Implement exponential backoff in the client:

```python
import time
import random

def request_with_backoff(url, max_retries=5):
    for attempt in range(max_retries):
        response = requests.get(url)
        if response.status_code != 429:
            return response
        wait = (2 ** attempt) + random.uniform(0, 1)
        time.sleep(wait)
    raise Exception("Max retries exceeded")
```

### ERR-2070: Internal Processing Error

An unexpected error occurred during request processing. This typically indicates a bug or an unhandled edge case.

**Log Example:**

```
2026-03-16 12:45:00 CRITICAL [order-service] ERR-2070
  request_id=req-b9a8c7d6
  endpoint=POST /api/v2/orders
  error=NullPointerException at OrderProcessor.java:142
  stack_trace="OrderProcessor.validate(OrderProcessor.java:142)
    -> OrderController.create(OrderController.java:58)
    -> DispatcherServlet.service(DispatcherServlet.java:1067)"
```

**Diagnostic Steps:**

1. Retrieve the full error details using the request ID:

```bash
curl -s "http://logging.internal:9200/app-logs-*/_search" \
  -H "Content-Type: application/json" \
  -d '{"query": {"term": {"request_id": "req-b9a8c7d6"}}}' | \
  jq '.hits.hits[]._source | {timestamp, level, message, stack_trace}'
```

2. Check if the error is a known issue by searching the issue tracker:

```bash
gh issue list --repo myorg/order-service --search "NullPointerException OrderProcessor"
```

3. If the error is reproducible, collect a diagnostic bundle:

```bash
curl -s http://localhost:8080/admin/diagnostic-bundle \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -o /tmp/diagnostic-bundle.zip
```
