## Overview

The 1000 series covers network-level and connectivity errors. These codes are generated by the connection manager and load balancer components.

---

## Error Code Reference

| Code | Severity | Description | Recommended Action |
|------|----------|-------------|--------------------|
| ERR-1000 | Critical | Connection pool exhausted | Increase pool size or investigate connection leaks |
| ERR-1001 | Error | TCP connection refused by upstream | Verify the upstream service is running and accepting connections |
| ERR-1002 | Error | TCP connection timed out | Check network routing and firewall rules between client and server |
| ERR-1003 | Warning | Connection reset by peer | The remote end closed the connection unexpectedly; check server logs |
| ERR-1010 | Error | DNS resolution failed | Verify DNS records exist and the resolver is reachable |
| ERR-1011 | Warning | DNS resolution slow (>500ms) | Check DNS server performance and consider local caching |
| ERR-1020 | Error | TLS handshake failed | Inspect certificate validity and trust chain |
| ERR-1021 | Warning | TLS certificate expires within 7 days | Renew the certificate before expiration |
| ERR-1022 | Error | TLS protocol version mismatch | Ensure both client and server support a common TLS version |
| ERR-1030 | Error | HTTP/2 protocol error | Check for proxy misconfiguration or incompatible middleware |
| ERR-1040 | Warning | Connection idle timeout reached | The connection was closed after exceeding the idle timeout; this is normal cleanup |
| ERR-1050 | Critical | All upstream hosts unhealthy | No healthy backend is available; investigate service health checks |
| ERR-1051 | Error | Upstream health check failed | A specific upstream host failed its health check |
| ERR-1060 | Warning | Connection retry initiated | Automatic retry is in progress; monitor for repeated occurrences |
| ERR-1070 | Error | Maximum redirects exceeded | The request followed too many redirects; check for redirect loops |

## Detailed Descriptions

### ERR-1000: Connection Pool Exhausted

This error occurs when the application has used all available connections in the pool and cannot open a new one.

**Log Example:**

```
2026-03-15 08:12:44 CRITICAL [conn-manager] ERR-1000
  pool_name=upstream-api
  active=100
  max=100
  waiting=47
  avg_checkout_ms=12400
```

**Diagnostic Steps:**

1. Check the current pool state:

```bash
curl -s http://localhost:8080/admin/pool-stats | jq '.'
```

Expected output:

```json
{
  "pool": "upstream-api",
  "active": 100,
  "idle": 0,
  "max": 100,
  "waiting": 47,
  "avg_wait_ms": 12400
}
```

2. Look for connection leaks by checking if connections are held for unusually long periods:

```bash
curl -s http://localhost:8080/admin/pool-stats/connections | \
  jq '.connections | sort_by(-.held_seconds) | .[0:5]'
```

3. Temporarily increase the pool size while investigating:

```bash
curl -X PUT http://localhost:8080/admin/pool-config \
  -H "Content-Type: application/json" \
  -d '{"max_connections": 200}'
```

### ERR-1050: All Upstream Hosts Unhealthy

This critical error means the load balancer has no healthy backends to route traffic to.

**Log Example:**

```
2026-03-15 09:30:00 CRITICAL [load-balancer] ERR-1050
  cluster=api-backend
  total_hosts=3
  healthy=0
  last_healthy_at=2026-03-15T09:28:12Z
```

**Diagnostic Steps:**

1. Check the health status of all upstream hosts:

```bash
curl -s http://localhost:8080/admin/upstreams/api-backend | jq '.hosts[] | {address, healthy, last_check}'
```

2. Test individual upstream hosts directly:

```bash
for host in api-1:8080 api-2:8080 api-3:8080; do
  echo -n "$host: "
  curl -s -o /dev/null -w "%{http_code}" "http://$host/health"
  echo
done
```

3. Review the health check configuration to ensure thresholds are not too aggressive:

```yaml
# health_check.yaml
interval: 10s
timeout: 5s
unhealthy_threshold: 3
healthy_threshold: 2
path: /health
expected_status: 200
```
