Six months of private beta on the relay turned up one recurring complaint:
two people editing different fields on the same row — say, a device's name
and its firmware version — would sync back and one edit would silently
overwrite the other, because conflict resolution operated on the whole row.
0.9.0 rewrites the log format so a change carries its column alongside its
value, and the merge step reconciles column by column.

<!-- more -->

```bash
$ vector sync peer relay.vector.sh --database devices.vec
resolved 3 conflicts: 3/3 were column-disjoint, applied automatically
0 conflicts required manual review
```

That column-disjoint case is the common one in practice — most simultaneous
edits touch different parts of a row — and now merges cleanly instead of
picking a winner and discarding the loser's write.

`.explain` was smaller, but it closes a gap every early user hit:

```sql
vector> .explain select * from readings where device_id = ? order by ts desc limit 20;
scan: index readings_device_ts (device_id, ts desc)
rows examined (estimate): 20
```

It never runs the query — only plans it — so it is safe to point at
production data without a `limit` clause you are afraid of.
