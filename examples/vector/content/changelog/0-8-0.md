`.vx` went through three incompatible syntax revisions between 0.5 and 0.7
while the team argued about whether migrations should read as SQL DDL or as
a smaller declarative diff format. 0.8.0 settles it in favor of plain SQL,
because every user we interviewed during the beta already knew SQL and
nobody wanted to learn a second schema language just to define tables.

<!-- more -->

```sql
-- 0001_init.vx
create table devices (
    id integer primary key,
    name text not null,
    battery_pct integer
);

create table readings (
    device_id integer references devices(id),
    ts integer not null,
    value real not null
);
```

`vector init` now writes a file close to this one into any new project, so
the first thing a new user sees is a real, working migration rather than an
empty file and a blank page in the manual. The frozen syntax also means a
migration written against 0.8.0 will still apply cleanly against every later
0.x release — a promise the earlier, experimental formats could not make.

The `--create` fix removes the last piece of ceremony: pointing
`vector schema apply` at a path that does not exist yet now creates the
database file and applies the first migration in one step.
