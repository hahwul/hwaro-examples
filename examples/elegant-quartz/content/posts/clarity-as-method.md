title = "Clarity as Method"
description = "Notes on writing interfaces that explain themselves."
date = "2024-06-08"
+++

Clarity is not a stylistic preference. It is a working method, applied at every stage of design and revision, that asks the same question repeatedly: will the next person who encounters this understand what it is asking of them?

The question sounds simple. In practice, it is unforgiving. A button labeled Submit may be clear in one context and ambiguous in another. A field labeled Name may be obvious to a designer and confusing to a user filling out a tax form. Clarity is contextual, and the only way to verify it is to put the work in front of someone who has not seen it before and watch them try to use it.

Most clarity problems come from compression. A label is shortened to fit a column. A confirmation message is tightened to a single line. A help text is removed because the page felt cluttered. Each of these decisions is defensible in isolation. Together, they produce interfaces that demand the user already know how the system works.

The remedy is not verbosity. Long explanations create their own problems, including the assumption that the user will read them. The remedy is the right number of words in the right place, which is a craft skill, developed over time, that resists shortcuts.

A useful test: read the interface aloud to someone unfamiliar with the product. Where they ask a question, the interface has failed to answer it. The cost of fixing the answer in the design is always lower than the cost of fielding the same question repeatedly in support.
