A targeted patch. The headline fix closes a rare but nasty chunk-save corruption
that could occur when the host was killed mid-flush; if you run on spot instances or
flaky power, apply this on sight.

<!-- more -->

## Fixed

- **Chunk-save corruption on abrupt shutdown.** The region writer used a
  rename-over-original strategy that left a truncated file if the process died
  between `write` and `rename`. It now writes to a temp file, `fsync`s, and renames
  atomically, so a half-written region can never replace a good one.

  ```bash
  # verify your existing regions after upgrading
  $ milepost world verify survival --repair
  scanned 4,192 regions · 0 corrupt · 3 orphaned tempfiles removed
  ```

- Fixed a crash when a plugin called `ctx.world.setBlock()` with coordinates in a
  chunk that had just been unloaded; the call now no-ops and logs at debug level.
- `milepost backup` no longer races the region writer — backups briefly quiesce the
  flush queue to capture a consistent snapshot.

## Upgrade notes

Pure patch, no config or API changes. After upgrading, run `milepost world verify
--repair` once per world to sweep up any orphaned temp files left by the old writer.
Rolling back to `2.2.0` is safe; regions written by `2.2.1` are byte-compatible.
